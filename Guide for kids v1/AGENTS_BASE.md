# AGENTS_BASE.md
Версия: v1.1
Назначение: переиспользуемые правила работы для переноса дизайна из Figma в production-ready код.

---

## 1) Миссия
Собирать UI из Figma так, чтобы результат был:
- визуально точным;
- стабильным на брейкпоинтах;
- безопасным для существующей логики проекта;
- удобным в поддержке после передачи.

Основной порог качества:
- соответствие структуре и поведению из Figma;
- отсутствие регрессий в незатронутых блоках;
- изменения легко проверить и откатить.

---

## 2) Приоритеты (если правила конфликтуют)
1. Не ломать рабочее поведение.
2. Оставаться в согласованном scope.
3. Сохранять визуальное соответствие Figma.
4. Переиспользовать текущую архитектуру до добавления новых паттернов.
5. Делать реализацию простой и поддерживаемой.

---

## 3) Scope и дисциплина изменений
### 3.1 Сначала scope
- Если задача локальная (например, только `Menu`), не менять несвязанные блоки.
- Небольшие соседние правки допустимы только как прямое следствие задачи.
- Любой расширенный рефакторинг требует явного согласования с пользователем.

### 3.2 Сначала переиспользование
Перед написанием нового кода проверить, можно ли расширить существующее:
- компоненты/блоки;
- CSS-токены и утилиты;
- текущую стратегию media queries;
- существующую JS-логику (dropdown, modal, i18n-хелперы).

Не переписывать рабочий блок только ради косметической унификации.

### 3.3 Пошаговая поставка
Работать небольшими change-set:
1. Проанализировать текущую реализацию.
2. Сформулировать краткий план.
3. Внедрить один логический change-set.
4. Провалидировать поведение.
5. Переходить к следующему change-set.

---

## 4) Figma как источник истины
Использовать Figma как канонический визуальный источник:
- Dev Mode;
- структура и значения через MCP;
- именованные фреймы/состояния;
- экспортированные ассеты.

Не использовать скриншоты как основной источник, если доступна структура слоев.

### 4.1 Конвенция названий фреймов
Ожидаемый формат:
`<Block> <State> [<Breakpoint> · <Theme>]`

Пример:
`Menu Default [1920 · Light]`

### 4.2 Состояния для проверки (если есть в дизайне)
- Default
- Hover
- Active
- Disabled
- Layout/resizing behavior

Фреймы `Layout` — это поведенческая спецификация, а не опциональная справка.

---

## 5) Требования к адаптиву
Минимальные контрольные точки:
- 1920
- 1280
- 768
- 375

Правила:
- не делать desktop-only реализацию;
- не допускать горизонтальный overflow, если он не задуман;
- сохранять ритм отступов и геометрию секций на каждом брейкпоинте.

---

## 6) Правила по ассетам
Предпочтительные форматы:
- иконки: SVG;
- растр: PNG/WebP;
- поддержка высокой плотности: @2x/@3x при необходимости.

При замене ассетов:
1. Подключить новый файл в разметке/стилях.
2. Проверить отображаемый размер и качество на Retina.
3. Убедиться, что старый ассет нигде больше не используется.
4. Удалять старый ассет только после проверки использования.

---

## 7) Интерактив и доступность
Для интерактивного UI как минимум проверить:
- поведение с клавиатуры и мыши/тачпада;
- видимые focus-состояния;
- корректные переходы состояний (default/hover/active/disabled);
- отсутствие поломок ARIA-label/role после изменений.

Для модалок:
- закрытие по клику в overlay;
- закрытие по `Esc`;
- закрытие по кнопке закрытия;
- корректное поведение фокуса.

---

## 8) Правила локализации
- Любой новый пользовательский текст должен идти через i18n.
- Не оставлять постоянные захардкоженные строки в feature-коде.
- Соблюдать единый стиль translation keys.
- Fallback-язык — страховка, а не замена полноценных переводов.

---

## 9) Стратегия браузерной валидации
### 9.1 Обязательно
- Chromium-based браузер (последний стабильный)
- Safari (последний стабильный desktop)

### 9.2 Условно обязательно
- Safari iOS для релиз-критичных или layout-чувствительных изменений.

Зоны повышенного риска в Safari:
- fixed/sticky позиционирование;
- scroll-контейнеры и clipping overflow;
- modal overlay;
- viewport units (`vh`/`dvh`);
- SVG и Retina-рендеринг;
- поведение flex min-size.

---

## 10) Definition of Done
Перед завершением задачи проверить:
- визуальный результат соответствует Figma в рамках запроса;
- поведение корректно на обязательных брейкпоинтах;
- нет регрессий в соседних блоках;
- i18n и доступность не нарушены;
- измененные файлы проходят lint/checks;
- ассеты и ссылки валидны.

---

## 11) Формат ответа перед внедрением
До внесения изменений показать:
1. План (краткий, по шагам)
2. Какие файлы будут изменены
3. Риски
4. Чеклист проверки

Формулировать кратко и прикладно.
